What is journalctl?
Banglish (বাংলা + English) গাইড — Linux logging system সম্পূর্ণ বোঝার জন্য
SECTION 01 journalctl কী?

journalctl হলো systemd-এর official log reading tool। তোমার Linux system এ যা কিছু হচ্ছে — service start হচ্ছে, error আসছে, kernel কিছু বলছে — সব কিছু systemd-journald নামের একটা daemon collect করে রাখে।

journalctl হলো সেই stored data পড়ার tool — filter করা যায়, search করা যায়, real-time stream করা যায়। একটাই tool দিয়ে system এর সব কিছু monitor করা যায়।

পুরনো vs নতুন
/var/log/syslog Plain text — search করতে grep লাগে, structured data নেই
journald Binary + indexed — fast search, metadata, JSON output, filters
SECTION 02 Architecture
┌─[Kernel] ┌─[Services] ┌─[Apps] │ /dev/kmsg │ stdout/stderr │ syslog() └──────┬─────┘──────┬──────┘────┬──┘ └────────────┴───────────┘ ↓ systemd-journald ↓ /var/log/journal/ (binary, persistent) /run/log/journal/ (RAM, volatile) ↓ journalctl ← তুমি এটা দিয়ে পড়ো
Kernel — hardware error, driver issue /dev/kmsg থেকে আসে
Services — nginx, sshd এর stdout/stderr সরাসরি journald এ যায়, কোনো file লেখার দরকার নেই
Apps — sd_journal_send() বা syslog() দিয়ে যেকোনো program log পাঠাতে পারে
SECTION 03 একটা Log Entry তে কী থাকে?

journalctl -o json দিলে প্রতিটা entry এরকম দেখায়:

{ "MESSAGE": "nginx started", "_SYSTEMD_UNIT": "nginx.service", "PRIORITY": "6", ← 0=emerg…7=debug "_PID": "1234", "_HOSTNAME": "my-server", "__REALTIME_TIMESTAMP": "1741..." ← microseconds }

SysWatch এই JSON parse করে MESSAGE, _SYSTEMD_UNIT, PRIORITY নিয়ে dashboard এ দেখায়। jq দিয়ে terminal এ filter করতে পারো: journalctl -o json | jq '.MESSAGE'

SECTION 05 SysWatch কীভাবে journalctl ব্যবহার করছে?
Browser কখনো directly journalctl চালাতে পারে না — security issue। তাই Node.js server trusted intermediary হিসেবে কাজ করছে।
তুমি filter select করো (unit, priority) ↓ Node.js → spawn('journalctl', ['-f', '-u', 'nginx', ...]) ↓ proc.stdout → line-by-line JSON parse ↓ SSE (text/event-stream) → HTTP push ↓ Browser EventSource → auto-reconnects ↓ তুমি screen এ real-time দেখছো
Unit বা priority change করলে — পুরনো journalctl process kill হয়, নতুন args দিয়ে আবার start হয়। এটাই তুমি Journal Logs page এ করছো।

SECTION 07 journalctl কোথা থেকে data নেয়?

Linux এ সব system logs একটা জায়গায় জমা হয় — systemd-journald নামের একটা service এটা manage করে। এটা একটা binary format এ store করে — তাই সরাসরি cat করলে কাজ হবে না, journalctl দিয়েই পড়তে হয়।

এই folders এ actual data আছে
ls /run/log/journal/ ls /var/log/journal/ # persistent হলে
তোমার programs/services
↓
systemd-journald ← সব logs collect করে
↓
/run/log/journal/ ← binary format এ store
↓
journalctl ← binary files পড়ে দেখায়
SECTION 08 journalctl কিভাবে stream করে?

journalctl -f দিলে দুইটা কাজ হয়: প্রথমে পুরনো কিছু logs দেখায়, তারপর নতুন log আসলে সাথে সাথে print করে। এটা সম্ভব হয় inotify দিয়ে — Linux kernel এর একটা feature।

নতুন log entry আসলো
↓
kernel inotify দিয়ে journald কে জানালো
↓
journald binary file এ লিখলো
↓
journalctl সেটা পড়ে terminal এ print করলো
💡 inotify কী? File polling (বারবার check করা) না — kernel event-driven ভাবে notify করে। File change হলে instantly জানায়। এই জন্যই journalctl -f এত fast।
journalctl -f -o json দিলে প্রতিটা entry একটা JSON line:
{"MESSAGE":"nginx started","_SYSTEMD_UNIT":"nginx.service","PRIORITY":"6","__REALTIME_TIMESTAMP":"1234567890"}
এই format টাই Node.js এর জন্য perfect — JSON.parse() করেই কাজ শুরু।
SECTION 09 Node.js এটা browser এ কিভাবে পাঠায়?

তিনটা ধাপে কাজ হয় —

ধাপ ১ — journalctl কে child process হিসেবে চালায়
Node.js — server side
const { spawn } = require('child_process');

// ঠিক terminal এ লেখার মতোই
// journalctl -f -o json
const proc = spawn('journalctl', ['-f', '-o', 'json']);
ধাপ ২ — stdout থেকে real-time পড়ে
proc.stdout.on('data', (chunk) => {
  // journalctl যখনই নতুন log print করে
  // Node.js এখানে পায় — real-time
  const json = chunk.toString();
});
ধাপ ৩ — SSE দিয়ে browser এ push করে
res.setHeader('Content-Type', 'text/event-stream');
// ↑ Browser কে বলছি — connection খোলা রাখো

proc.stdout.on('data', chunk => {
  res.write(`data: ${chunk}\n\n`);
  //          ↑ SSE format
  // "data: " দিয়ে শুরু, "\n\n" দিয়ে শেষ
});

req.on('close', () => proc.kill());
// Browser disconnect করলে process kill
Browser side — মাত্র এটুকুই লাগে
const source = new EventSource('/api/logs/stream');

source.onmessage = (event) => {
  const log = JSON.parse(event.data);
  // log.MESSAGE, log._SYSTEMD_UNIT পাওয়া গেলো
  console.log(log.MESSAGE);
};

Real example — nginx এর দুই ধরনের log
# service start/stop/crash — journalctl তে
journalctl -u nginx
# Mar 07 systemd: Started nginx

ধাপ ২ — journald তিন source থেকে নেয়
① /dev/log socket
পুরনো programs যেগুলো syslog() call করে
② systemd journal socket
services এর stdout/stderr সরাসরি
③ kernel /dev/kmsg
hardware, driver, network messages — same as dmesg
ধাপ ৩ — metadata add করে store
{
 MESSAGE: "worker started",
 _SYSTEMD_UNIT: "nginx.service",
 _PID: "1234",
 _EXE: "/usr/sbin/nginx",
 PRIORITY: "6",
 __CURSOR: "s=abc...xyz",
 _BOOT_ID: "abc123..."
}
Program কিছু না করলেও journald নিজেই সব metadata add করে।
ধাপ ৪ — Filter কিভাবে কাজ করে?
journalctl -u nginx internally যা করে: binary files খোলে → index এ _SYSTEMD_UNIT == "nginx.service" খোঁজে → matching entries বের করে → timestamp sort → দেখায়।
# Multiple filters একসাথে
journalctl -u nginx -p err --since "1h ago"

# Specific PID

# Kernel only (= dmesg)
journalctl -k

# Previous boot
journalctl -b -1
Plain text vs Binary — কেন binary better?
Plain text grep করতে হয়, slow, metadata আলাদা রাখতে হয়
Binary indexed, instant filter, metadata একসাথে, compressed

📊 Metrics — /proc files
CPU usagecopy
cat /proc/stat
Linux exposes raw CPU tick counters here. Read it twice 500ms apart, compute: (total−idle)/total×100. This is how top and htop work internally.
Fieldsuser, nice, system, idle, iowait, irq, softirq, steal
idle+iowait= "doing nothing" ticks
Memorycopy
cat /proc/meminfo
All memory stats in kB. Use MemAvailable (not MemFree) — Linux caches files in unused RAM, so MemFree appears low but memory IS available.
Load averagecopy
cat /proc/loadavg
5 values: 1min, 5min, 15min load, running/total procs, last PID. Load of 1.0 on 1 CPU = 100% saturated. On 4 CPUs, saturated = 4.0.
Network I/Ocopy
cat /proc/net/dev
Cumulative RX/TX byte counters per interface. Compute KB/s by reading twice, dividing byte-difference by elapsed time.
⬡ Node.js reads /proc directly
const fs = require('fs'); // CPU: read twice, diff the ticks const line = fs.readFileSync('/proc/stat') .toString().split('\n')[0]; // Memory: parse key:value lines const mem = fs.readFileSync('/proc/meminfo').toString(); const total = mem.match(/MemTotal:\s+(\d+)/)?.[1];
💡 Why no npm packages? Reading /proc is a raw file read — almost zero overhead. The kernel writes these files live in memory, so fs.readFileSync is perfect and fast.

⚙ systemctl Commands
List all servicescopy
systemctl list-units --type=service --all --output=json
Lists every service unit systemd knows. --all includes stopped ones. --output=json makes it machine-readable for Node.js.
--type=serviceOnly .service units (not timers, sockets)
--allInclude inactive and failed units too
--no-pagerDon't pipe through less — needed in scripts
Service detailscopy
systemctl show nginx.service \ --property=MainPID,MemoryCurrent,ActiveState
Machine-readable properties — no parsing needed. Returns KEY=VALUE lines. Much faster than parsing systemctl status output.
Control a servicecopy
systemctl start|stop|restart nginx.service
Requires root. In Node.js use execP('systemctl restart nginx'). Secure it with a sudoers rule — never give the web process unrestricted sudo.
⬡ Node.js — systemctl JSON
const { stdout } = await execP( `systemctl list-units --type=service \ --all --output=json --no-pager` ); const units = JSON.parse(stdout); // each unit: { unit, load, active, sub, description }
💡 sudoers tip: Add www-data ALL=(root) NOPASSWD: /bin/systemctl restart nginx to allow only that one command — nothing else.

⬡ Node.js SSE stream
// Server-Sent Events: real-time log stream app.get('/api/logs/stream', (req, res) => { res.writeHead(200, { 'Content-Type': 'text/event-stream', 'Cache-Control': 'no-cache', }); const proc = spawn('journalctl', ['-f', '-o', 'json']); proc.stdout.on('data', d => res.write(`data: ${d}\n\n`)); req.on('close', () => proc.kill()); }); // Browser side: const es = new EventSource('/api/logs/stream'); es.onmessage = e => { const entry = JSON.parse(e.data); // entry.MESSAGE, entry._SYSTEMD_UNIT, etc. };
💡 SSE vs WebSocket: SSE is one-way (server→browser), auto-reconnects, and works over plain HTTP. Perfect for log streaming. Use WebSockets only if you need bidirectional messaging.

journalctl কী?

journalctl হলো systemd-এর official log reading tool। তোমার Linux system এ যা কিছু হচ্ছে — service start হচ্ছে, error আসছে, kernel কিছু বলছে — সব কিছু systemd-journald নামের একটা daemon collect করে রাখে। আর journalctl সেই data পড়ার CLI tool।

পুরনো system এ /var/log/syslog ছিল plain text file। কিন্তু journald binary format এ store করে — faster search, structured metadata, automatically indexed। প্রতিটা log entry একটা JSON object এর মতো — timestamp, PID, unit name, priority সব আলাদা field হিসেবে থাকে।

Architecture — কীভাবে কাজ করে?
╔══════════════════════════════════════╗ ║ [Kernel] [Services] [Apps] ║ ║ ↓ ↓ ↓ ║ ║ systemd-journald ← সব এখানে আসে ║ ║ ↓ ║ ║ /var/log/journal/ (binary files) ║ ║ ↓ ║ ║ journalctl ← তুমি এটা দিয়ে পড়ো ║ ╚══════════════════════════════════════╝
① Kernel logs — /dev/kmsg থেকে journald নেয়। Hardware error, driver issue এখানে আসে।
② Service logs — nginx, sshd এর মতো service এর stdout / stderr সরাসরি journald এ যায়। কোনো file লেখার দরকার নেই।
③ App logs — যেকোনো program sd_journal_send() বা syslog() দিয়ে log পাঠাতে পারে।
④ journald সব collect করে binary journal files এ লেখে। /var/log/journal/ — persistent (reboot এর পরেও থাকে)। /run/log/journal/ — volatile (RAM এ, reboot এ মুছে যায়)।
একটা Log Entry তে কী থাকে?

-o json দিলে প্রতিটা entry এরকম দেখায়:

{ "MESSAGE": "Started nginx service", "_SYSTEMD_UNIT": "nginx.service", "PRIORITY": "6", ← info "_PID": "1234", "_HOSTNAME": "my-server", "__REALTIME_TIMESTAMP": "1741..." ← microseconds }

SysWatch এই JSON parse করে MESSAGE, _SYSTEMD_UNIT, PRIORITY নিয়ে তোমাকে দেখায়। Raw journalctl output এর চেয়ে অনেক বেশি structured।

Priority Levels — কোনটা কতটা গুরুত্বপূর্ণ?
0 emerg — System crash হয়ে গেছে, কিছুই কাজ করছে না
1 alert — এখনই action নিতে হবে
2 crit — Critical failure (hardware/software)
3 err — Error — সবচেয়ে বেশি দেখবে, fix করতে হবে
4 warning — Warning — এখনই problem না, কিন্তু নজর রাখো
5 notice — Normal কিন্তু noteworthy কিছু হয়েছে
6 info — Normal operational messages
7 debug — Verbose debug info — সাধারণত filter করে রাখো
💡 টিপস: -p err দিলে 0–3 (emerg, alert, crit, err) সব দেখায় — শুধু error না। জানো — lower number = বেশি severe। এটা syslog এর পুরনো convention।
SysWatch কীভাবে journalctl ব্যবহার করছে?
SysWatch Node.js এ child_process.spawn() দিয়ে journalctl চালায় এবং output কে browser এ real-time push করে। Flow টা এরকম:
journalctl -f -o json ← child process ↓ proc.stdout (stream) ← line by line JSON ↓ SSE event: data: {...} ← HTTP text/event-stream ↓ Browser EventSource ← auto reconnects ↓ DOM update ← তুমি screen এ দেখছো
Browser কখনো directly journalctl চালাতে পারে না — security issue। তাই Node.js server টা trusted intermediary হিসেবে কাজ করছে। Unit বা priority filter change করলে — পুরনো connection বন্ধ করে, নতুন args দিয়ে আবার journalctl start হয়।

journalctl -p err -f — শুধু errors, live

List processescopy
ps aux --sort=-%cpu
Snapshot of all processes sorted by CPU. Unlike top it exits immediately — easy to parse. SysWatch reads /proc/PID/* directly for better control.
aAll users' processes
uShow username, CPU%, MEM%
xInclude processes without a terminal
--sort=-%memSort by memory descending
/proc/PID filescopy
cat /proc/1234/comm # process name cat /proc/1234/status # VmRSS, Uid, state cat /proc/1234/stat # CPU ticks (field 14,15) cat /proc/1234/cmdline # full command line
Every running process has a directory in /proc/PID/. Reading these files is how SysWatch gets live data without any external tool.
Kill a processcopy
kill -15 <PID> # SIGTERM (graceful) kill -9 <PID> # SIGKILL (force)
Always try SIGTERM (15) first — it lets the process clean up. SIGKILL (9) can't be caught or ignored; the kernel terminates it immediately.

▶ systemd-analyze Commands
Total boot timecopy
systemd-analyze
Shows total time split: firmware + bootloader + kernel + userspace. Quick benchmark for your server's startup speed.
Per-service blamecopy
systemd-analyze blame
Lists every service sorted by how long it took to initialize. Find what's slowing your boot. Use --json=short for Node.js parsing.
Critical dependency chaincopy
systemd-analyze critical-chain
The slowest path from kernel to multi-user.target. Like a flamegraph for boot — shows which dependency chain determined total boot time.
Generate SVG flamegraphcopy
systemd-analyze plot > boot.svg
Generates a beautiful SVG timeline of the entire boot sequence. Open in a browser.
⬡ Node.js — parse blame
const { stdout } = await execP( 'systemd-analyze blame --json=short' ); // returns array of { name, time } objects const blame = JSON.parse(stdout); // blame[0] = slowest service
💡 Reduce boot time: Disable services you don't need: systemctl disable bluetooth.service. Use systemctl mask to prevent a service from ever starting.
